## 1. What is i18n?

**i18n** (internationalization) is the process of designing an app so it can be adapted to different languages and regions. The name comes from "i" + 18 letters + "n" in the word "internationalization".

### Without i18n

```tsx
// ❌ Hard-coded English
function HabitCard() {
  return (
    <View>
      <Text>Check in</Text>
    </View>
  );
}
```

### With i18n

```tsx
// ✅ Dynamic translation
function HabitCard() {
  const { t } = useI18n();
  return (
    <View>
      <Text>{t.checkIn}</Text>  // "Check in" in English, "打卡" in Chinese
    </View>
  );
}
```

## 2. Detecting Device Language

We use `expo-localization` to detect the device's language setting:

```tsx
import { getLocales } from 'expo-localization';

const locales = getLocales();
console.log(locales[0].languageCode); // 'en', 'zh', 'ja', etc.
```

The `getLocales()` function returns an array of locales the user has configured (primary first). We extract `languageCode` from the first one.

### The Translation Files

**English** — `src/locales/en.ts`:

```tsx
export const en = {
  habits: 'Habits',
  settings: 'Settings',
  newHabit: 'New Habit',
  save: 'Save',
  checkIn: 'Check in',
  checked: 'Checked',
  // ... 26 keys total
};
```

**Chinese** — `src/locales/zh.ts`:

```tsx
export const zh = {
  habits: '习惯',
  settings: '设置',
  newHabit: '新建习惯',
  save: '保存',
  checkIn: '打卡',
  checked: '已打卡',
  // ... same 26 keys
};
```

Both files have **exactly the same keys**. This is enforced by TypeScript — if one is missing a key, the type system catches it.

## 3. React Context for Global Access

We need the current language and translation function to be available everywhere in the app. **React Context** is the right tool for this.

### What is Context?

Context lets you pass data through the component tree without manually passing props at every level.

```
Without Context:
App → Tabs → HabitsScreen → HabitCard → CheckButton
          ↘ SettingsScreen
                 (language props would need to pass through every level)

With Context:
<Context.Provider value={{ lang: 'zh', t: translations['zh'] }}>
  App → Tabs → HabitsScreen → HabitCard → CheckButton  (all call useI18n() directly)
          ↘ SettingsScreen                               (all call useI18n() directly)
</Context.Provider>
```

### The I18nProvider

Located at `src/contexts/i18n.tsx`:

```tsx
import { createContext, useContext, useEffect, useState, type ReactNode } from 'react';
import { useSQLiteContext } from 'expo-sqlite';
import { getLocales } from 'expo-localization';
import { en } from '@/locales/en';
import { zh } from '@/locales/zh';

type Lang = 'en' | 'zh';
type Translations = Record<string, string>;
const translations: Record<Lang, Translations> = { en, zh };

type I18nContextType = {
  lang: Lang;
  t: Translations;
  setLang: (lang: Lang) => Promise<void>;
};

const I18nContext = createContext<I18nContextType | null>(null);

function getSystemLang(): Lang {
  const code = getLocales()[0]?.languageCode;
  return code === 'zh' ? 'zh' : 'en';
}

export function I18nProvider({ children }: { children: ReactNode }) {
  const db = useSQLiteContext();
  const [lang, setLangState] = useState<Lang>(getSystemLang);

  // On mount, load saved language from database
  useEffect(() => {
    db.getFirstAsync<{ value: string }>(
      'SELECT value FROM settings WHERE key = ?',
      'language'
    ).then((row) => {
      if (row && (row.value === 'en' || row.value === 'zh')) {
        setLangState(row.value as Lang);
      }
    });
  }, [db]);

  // setLang persists the choice
  const setLang = async (newLang: Lang) => {
    setLangState(newLang);
    await db.runAsync(
      'INSERT OR REPLACE INTO settings (key, value) VALUES (?, ?)',
      'language',
      newLang
    );
  };

  return (
    <I18nContext.Provider value={{ lang, t: translations[lang], setLang }}>
      {children}
    </I18nContext.Provider>
  );
}

export function useI18n() {
  const ctx = useContext(I18nContext);
  if (!ctx) throw new Error('useI18n must be used within I18nProvider');
  return ctx;
}
```

### Breaking Down the Code

**Step 1: Create the context**
```tsx
const I18nContext = createContext<I18nContextType | null>(null);
```
This creates a context with a default value of `null`. The `I18nContextType` is the shape of data we'll provide: `{ lang, t, setLang }`.

**Step 2: Define the provider**
```tsx
<I18nContext.Provider value={{ lang, t: translations[lang], setLang }}>
  {children}
</I18nContext.Provider>
```
The provider makes `{ lang, t, setLang }` available to every descendant component.

**Step 3: Consume with a hook**
```tsx
export function useI18n() {
  const ctx = useContext(I18nContext);
  if (!ctx) throw new Error('useI18n must be used within I18nProvider');
  return ctx;
}
```
The `useI18n` hook abstracts `useContext(I18nContext)` and adds a safety check — if someone forgets to wrap the app in `I18nProvider`, they get a clear error message instead of a cryptic runtime crash.

### Why Context + SQLite?

We store the language preference in the `settings` SQLite table:

```sql
INSERT OR REPLACE INTO settings (key, value) VALUES ('language', 'zh')
```

When the app starts:
1. `I18nProvider` mounts
2. Defaults to system language (`getSystemLang()`)
3. Then reads from SQLite — if the user previously chose a language, it overrides the default
4. The UI updates automatically

When the user switches language:
1. `setLang('zh')` is called
2. State updates: `setLangState('zh')`
3. The context re-renders with new translations
4. Persists: `INSERT OR REPLACE INTO settings ...`

## 4. Using i18n in Components

Components consume the hook:

```tsx
import { useI18n } from '@/contexts/i18n';

function SettingsScreen() {
  const { t, lang, setLang } = useI18n();

  return (
    <View>
      <Text>{t.language}</Text>         {/* "Language" or "语言" */}
      <Text>{lang === 'en' ? t.english : t.chinese}</Text>

      <Button onPress={() => setLang('zh')}>
        <Text>{t.chinese}</Text>
      </Button>
      <Button onPress={() => setLang('en')}>
        <Text>{t.english}</Text>
      </Button>
    </View>
  );
}
```

### In Tab Titles

The tab names are also localized:

```tsx
// src/app/(tabs)/_layout.tsx
export default function TabLayout() {
  const { t } = useI18n();

  return (
    <Tabs>
      <Tabs.Screen
        name="index"
        options={{
          title: t.habits,         // "Habits" or "习惯"
        }}
      />
      <Tabs.Screen
        name="settings"
        options={{
          title: t.settings,       // "Settings" or "设置"
        }}
      />
    </Tabs>
  );
}
```

When the user switches language, `t` changes → the context re-renders → the tab titles update.

## 5. Provider Nesting Order

The providers are nested in `_layout.tsx`:

```tsx
<DatabaseProvider>      ← Provides SQLite context
  <I18nProvider>        ← Uses SQLite context for persistence
    <Stack>             ← Screens use I18n context
  </I18nProvider>
</DatabaseProvider>
```

**Order matters**: `I18nProvider` calls `useSQLiteContext()` internally. If you swapped them, `I18nProvider` would throw because the SQLite context wouldn't exist yet.

## 6. Adding a New Translation

1. Add the key to both `en.ts` and `zh.ts`
2. Use it in components: `t.yourNewKey`
3. TypeScript catches typos automatically

```tsx
// en.ts
export const en = {
  // ... existing keys
  achievements: 'Achievements',   // New key
};

// zh.ts
export const zh = {
  // ... existing keys
  achievements: '成就',           // Same new key
};

// Usage
<ThemedText>{t.achievements}</ThemedText>
```
